Executive Summary

The application is a cross-platform file-transfer experience. The visible Flutter client authenticates users, lets them select a file, chooses a short expiry window, uploads through the API, and returns a copyable share link. The backend is a Python FastAPI service that separates authentication, upload, share, and download responsibilities behind typed Pydantic schemas and SQLAlchemy data models.

Core Pattern Metadata in PostgreSQL, binaries in S3-compatible storage, transient upload state in Redis.
Primary Stack Flutter, Riverpod, Dio, FastAPI, SQLAlchemy, PostgreSQL, Redis, Celery, MinIO/S3, Nginx.
Current Client Path The Flutter UI calls /api/upload/simple; chunked upload endpoints are implemented on the backend.

Client Application Layer

Flutter architecture and the UI-to-API contract used by the app.

Application shell

lib/main.dart wraps the app in ProviderScope, reads authProvider, and selects one of three first screens: a loading view, UploadScreen, or LoginScreen. The theme is defined centrally in AppTheme.dark().

  • State: Riverpod StateNotifier providers expose immutable state objects.
  • Persistence: shared_preferences stores the bearer token locally.
  • Navigation: login/register replace the stack with the upload screen after success.

API service

lib/services/api_service.dart creates one Dio client with a base URL from API_BASE_URL at compile time, falling back to a local network IP. An interceptor injects Authorization: Bearer when a saved token exists.

  • Auth calls: /api/auth/login and /api/auth/register.
  • Upload call: /api/upload/simple with multipart file and expiry minutes.
  • Error handling: maps common API status codes to user-facing messages.

Authentication state

AuthNotifier checks for a saved token on startup, performs login/register requests, writes the returned JWT, and clears it on logout. This keeps the UI simple: screens react to the provider state.

File transfer state

FileNotifier tracks selected path, loading status, generated link, and expiry minutes. The UI bounds expiry between 10 and 60 minutes, then calls the API service to upload and receive the share URL.

Current client implementation boundary

The backend implements chunked upload endpoints, but the current Flutter upload workflow uses the simple single-file endpoint. This is important for deployment planning because Nginx currently sets client_max_body_size 10M; larger files should use the chunked flow or adjust edge limits.

Backend API Layer

FastAPI route groups, service responsibilities, and request validation.

FastAPI app lifecycle

app.main starts a FastAPI application, applies CORS, mounts /api, verifies database tables through SQLAlchemy metadata, and attempts object-storage bucket initialization during lifespan startup.

Route groups

app.api.router mounts auth, upload, share, and download routers. Each route uses Pydantic schemas for input/output and FastAPI dependencies for authenticated user and database session injection.

Primary Workflows

End-to-end behavior for authentication, upload, share-link creation, and download.

Visible Flutter upload flow

1
User authenticates The client submits email/password and stores the JWT returned from /api/auth/login.
2
User selects file and expiry file_picker supplies a local path; expiry is constrained to 10-60 minutes.
3
Client posts multipart upload /api/upload/simple uploads bytes to object storage, creates a file row, and creates a share row.
4
Share URL returns to UI The response includes link and expiry_time; the UI displays a copy button.

Backend chunked upload capability

1
Start session /api/upload/start validates size, calculates chunks, creates a Redis hash, and sets TTL.
2
Upload chunks /api/upload/chunk validates owner/index and stores each chunk at an S3 key.
3
Finalize file /api/upload/finalize verifies completeness and assembles the final object using S3 multipart upload.
4
Create metadata The route writes a files record with the final storage key and user ownership.

Share-link policy

share_service validates file ownership, generates a unique token, computes expiry from minutes or hours, optionally hashes a password, and stores download limits and counters in the shares table.

  • Default expiry is configured as 72 hours.
  • Minute-based expiry supports 10-60 minutes.
  • Protected links use bcrypt-hashed passwords.

Download behavior

download_service validates token state, expiry, password, active flag, and download limit. It then logs the request, increments download count, and either redirects to a presigned object URL or streams the object directly with HTTP range support.

  • GET /api/download/{token} returns a 307 redirect.
  • GET /api/download/{token}/direct streams bytes.
  • HEAD /api/download/{token} exposes size and range support.

Data Model and Storage

Persistence model and object-storage ownership strategy.

Database responsibilities

PostgreSQL stores identities, ownership, object-storage keys, share policy, and audit data. The async SQLAlchemy session dependency commits successful request work and rolls back exceptions.

  • Users: unique email, password hash, active flag.
  • Files: original name, generated name, size, MIME type, S3 key, owner.
  • Shares: token, expiry, optional password, limits, active flag.
  • Downloads: file/share references plus client IP and user agent.

Object-storage strategy

The storage service uses boto3 against S3 or MinIO. Chunks are stored under uploads/{user_id}/{upload_id}/chunks/{index}. Final files are assembled into uploads/{user_id}/{upload_id}/{filename}.

  • Bucket creation is attempted during app startup and before uploads.
  • Multipart copy avoids loading every chunk into backend memory.
  • Presigned URLs expire after one hour for download redirects.
  • Range reads support resumable direct downloads.
Schema migration note

The checked-in Alembic initial migration is currently a placeholder with no generated DDL. At runtime, Base.metadata.create_all verifies and creates tables during FastAPI startup. For production, explicit Alembic migrations should replace startup schema creation.

Security, Reliability, and Operations

Security controls and operational mechanisms directly present in source code.

Security controls

  • Password hashing: bcrypt via Passlib for user passwords and optional share-link passwords.
  • JWT authentication: HS256 bearer tokens with configurable expiry, defaulting to 60 minutes.
  • Ownership checks: upload sessions and share creation validate current-user ownership.
  • Public download gates: token state, active flag, expiry, password, and download limit are validated.
  • Filename handling: utility functions sanitize names and generate storage-safe unique identifiers.

Reliability controls

  • Database sessions: request-scoped async sessions commit, rollback, and close predictably.
  • Redis TTL: upload sessions expire after the configured TTL and reset on chunk activity.
  • Chunk cleanup: cancel and scheduled Celery tasks remove stale upload artifacts.
  • Range downloads: direct streaming supports HTTP Range headers for resume-capable clients.
  • Object-store retries: boto3 client config enables adaptive retry behavior.

Deployment topology

filesharingbackend/docker-compose.yml defines backend, PostgreSQL, Redis, MinIO, Nginx, Celery worker, and Flower services on a bridge network. The backend is served by Uvicorn on port 8000.

  • Backend: 8000
  • Nginx: 80
  • PostgreSQL: 5432
  • Redis: 6379
  • MinIO API/console: 9000 / 9001
  • Flower: 5555

Nginx edge behavior

nginx.conf proxies /api/ to the backend, forwards client headers, exposes /health, applies 100r/m per-IP rate limiting, and configures extended proxy timeouts for file-transfer requests.

  • client_max_body_size 10M is compatible with 5 MB chunks plus buffer.
  • Rate limit burst is set to 20 requests with no delay.
  • Proxy read/send timeout is 300 seconds for large transfers.

Background processing

Celery uses Redis as broker and result backend. Scheduled tasks deactivate expired share links every hour and clean up nearly expired in-progress upload sessions every 30 minutes.

Environment configuration

pydantic-settings centralizes app, database, Redis, object storage, file-size, expiry, and CDN settings. Sensitive values such as secret keys and access keys are expected from environment variables.
